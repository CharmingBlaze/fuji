# raylib

Fuji bindings for **raylib 5.x** (generated by wrapgen). The **official Fuji SDK** ships this folder, the C glue, and **vendored raylib** under `third_party/raylib_static/stage/` so you can build games without installing LLVM or raylib separately (static link uses the bundled `libraylib.a`; Windows also includes `raylib.dll` in `lib/` for dynamic setups).

**Full guide:** [docs/guides/raylib.md](../../docs/guides/raylib.md) (include paths, `FUJI_NATIVE_SOURCES`, DLLs, Linux ARM64).

## Quick start

**1.** Include the bindings (path is resolved from the SDKâ€™s `wrappers/` tree â€” see the guide):

```fuji
#include "raylib/raylib.fuji"
```

**2.** Point native build at the glue file:

```powershell
set FUJI_NATIVE_SOURCES=wrappers\raylib\wrapper.c
```

```bash
export FUJI_NATIVE_SOURCES=wrappers/raylib/wrapper.c
```

**3.** Build from the SDK root (or set `FUJI_RAYLIB_STAGE` / work under a layout that still finds `wrappers/` and `third_party/`):

```powershell
fuji build mygame.fuji -o mygame.exe
```

If the shipped **`third_party/raylib_static/stage`** tree is present, Fuji adds include + static library flags automatically (`FUJI_USE_VENDORED_RAYLIB`, default on).

**4.** **Windows:** If your executable needs **`raylib.dll`**, copy `third_party/raylib_static/stage/lib/raylib.dll` next to the `.exe` or add it to **`FUJI_BUNDLE_FILES`** for `fuji bundle`.

**5.** **Manual link** (no vendored tree, or extra libs): set **`FUJI_LINKFLAGS`** to your `-I`, `-L`, and `-lraylib` (and OS frameworks on macOS). See [docs/wrappers.md](../../docs/wrappers.md).

## Troubleshooting

| Issue | What to check |
|--------|----------------|
| Undefined symbol in `wrapper.c` | `FUJI_NATIVE_SOURCES` includes `wrapper.c`; `FUJI_LINKFLAGS` / vendored `stage/` finds raylib. |
| Could not find `raylib/raylib.fuji` | Run with SDK `fuji` so `wrappers/` is next to the compiler, or set **`FUJI_PATH`** to the SDK root. |
| Crash on start (Windows) | Missing **`raylib.dll`** next to the exe â€” copy from `stage/lib/`. |
